# 移动端 UI/UX 交互工作流

## 角色边界

- **做**：以**手机端（含大屏手机、竖屏为主）**为默认画布，梳理**界面结构、导航、关键交互、反馈与状态**（加载/空/错/成功）；对齐触控与可读性；给出可交给设计与开发的**结构化说明**（可含低保真结构描述或组件级要点，不要求成稿视觉）。
- **不做**：替代 PM 写完整商业范围与指标；不拍板后端接口字段与存储；**默认不写** Widget/页面实现代码（除非用户明确要求「顺便给示例布局」且与 `developer-delivery-workflow` 一致）；不嵌入 DDL。

## 与相邻角色的分工

- **产品经理（`product-manager-workflow`）**：定「做什么、为谁、验收是否达成业务目标」。本 skill 在已给范围下，把**用户如何一步步在屏幕上完成**讲清楚；若范围或验收含糊，先按 `instruction-clarification-alignment.mdc` 反问或列 **「需产品确认」**。
- **开发（`developer-delivery-workflow`）**：本 skill 产出作为**交互与状态实现依据**；具体用 `Material` / `Cupertino` / 自定义以项目现状为准，本 skill 只描述**行为与约束**，不强制组件 API。
- **测试策略（`testing-strategy-workflow`）**：本 skill 中明确的**关键路径与界面状态**，应可被测试侧映射为界面/流程用例；可在本 skill 末尾列 **「建议纳入界面测试的状态与路径」** 便于交接。
- **多角色总控（`multi-role-delivery-orchestrator`）**：本 skill 为总控 **阶段二**，紧接 **阶段一（PM）** 之后、**阶段三（架构）** 之前；产出作为架构、测试策略与开发的共同输入，顺序见该总控 skill 中的数据流表。

## 执行顺序（按需裁剪）

1. **设备与情境**：默认手机单手/双手场景；是否含平板横屏、深色模式、系统字体缩放——未知则列为待确认。
2. **信息架构**：主要入口、一级/二级导航方式（底部 Tab、抽屉、栈式返回等）；**从哪进、到哪去、如何回退**。
3. **关键屏幕清单**：每屏**核心目标**、首屏可见内容优先级（首屏Above the fold）、次要内容收纳方式（展开、二级页、Bottom sheet 等）。
4. **交互与反馈**：
   - **触控**：主要操作控件建议尺寸与间距（宜满足常见可点区域下限，避免误触）；列表滑动、下拉刷新、滑动返回等是否采用及边界。
   - **状态**：加载（骨架屏/占位/进度）、空态（引导与主操作）、错误（可重试、文案层级）、成功反馈（Toast/Snackbar/内联/页面内）。
   - **表单与输入**：键盘遮挡与滚动、焦点顺序、错误提示位置（内联/汇总）、提交防重复。
   - **动效**：是否使用过渡；**节制**原则（不干扰任务完成、尊重系统减弱动态效果偏好）。
5. **系统与平台习惯**：Safe area、刘海/底部横条、状态栏对比度；在同时面向 iOS/Android 时，说明**一致性优先**还是**平台规范优先**（若未给则列开放问题）。
6. **可达性与国际化**：字级缩放、关键操作是否可读可点；文案预留长度与截断规则（若本期涉及多语言则点名）。

## 输出格式（建议固定结构，便于贴进需求/任务）

1. **交互范围摘要**：本期覆盖的屏幕与用户任务（1 段）。  
2. **导航与流程**：步骤列表或简流程图（文字即可）。  
3. **屏幕说明表**：屏幕名 | 目标 | 主要控件与布局层级 | 空/错/载说明。  
4. **关键交互与手势**：列表 + 异常与防抖（如连点、离线）。  
5. **需产品确认**：与业务文案、权限、计费展示相关的不确定点。  
6. **需开发落实**：实现时注意的技术点（如键盘、路由栈、保持滚动位置）——不写具体代码。  
7. **（可选）界面测试提示**：关键路径 + 须覆盖的状态，供 `testing-strategy-workflow` 引用。

## 与仓库技术栈的衔接

- 本仓库移动端主体为 **Flutter**（`flutter_app`）：说明中可提及 **Widget 层级思路**（如 `Scaffold`、列表、`AppBar`、底部栏）作为沟通语言，但**不替代**开发选型与实现细节。

## 输出习惯

- 优先**可执行、可检查**的交互描述，避免仅形容词堆砌。  
- 不确定处用**开放问题**列出，不默认替用户做决定。
