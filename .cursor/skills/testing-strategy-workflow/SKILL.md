# 测试策略与验收对齐工作流

## 角色边界

- **做**：对照本期行为变更，判断须覆盖的测试层级（单元 / API / 界面）；把验收标准翻译成可检查用例；标优先级与范围；写明建议运行方式与自动化缺口登记；对齐 `new-feature-testing-requirement.mdc` 与 `multi-feature-incremental-test.mdc`。
- **不做**：默认**不编写**测试实现代码与生产业务代码（用例描述、清单、表格、命令建议除外）；不修改产品范围；不替架构做技术选型；不在应用内嵌 DDL。

## 与规则的对齐（须遵守）

- **三类测试**：凡新接口、新页面/流程、新领域逻辑、新数据契约等实质扩展，须在同一交付中规划 **单元、API、界面** 三类中**本期触达项**的覆盖策略；轻量变更按 `new-feature-testing-requirement.mdc`「轻量变更例外」处理并**写明降级原因**。
- **可执行性**：API 与界面层须以**可自动执行**的测试资产为目标；不得以「仅 Postman 截图」作为唯一方案，除非任务中已约定例外与替代验收方式。
- **增量交付**：多功能批次时，按功能切片给出**每片的测试范围**，避免堆到最后再测；与 `multi-feature-incremental-test.mdc` 一致。
- **环境缺口**：某类测试暂无法自动化时，须在输出中写**原因、补救计划、本期手工验收步骤**，不得静默省略。

## 执行顺序（按需裁剪）

1. **输入确认**：本期需求摘要、验收标准（或 PM 产出）、是否涉及新/变 API、是否涉及用户可见流程、是否涉鉴权/权限、是否涉 schema（仅策略层面：提醒 DDL 与初始化由运维执行，见 `database-ddl-policy.mdc`）。
2. **变更归类**：判断属于「实质扩展」还是「轻量变更」；若信息不足，先按 `instruction-clarification-alignment.mdc` 简短反问再定级。
3. **分层映射**：
   - **单元**：纯函数、领域规则、校验、服务层核心分支与边界；建议落点：Flutter 为 `flutter_app/test/`（及子项目同类目录）；其他语言子项目遵循各自 `tests/` 或仓库惯例。
   - **API**：成功路径、主要错误码、鉴权/权限（若适用）；说明预期测试形态（集成/契约等），不写具体框架绑架句除非仓库已统一。
   - **界面**：关键用户路径；若本期涉及加载/空态/失败态，须至少在策略中点名其一；Flutter 侧可指向 `integration_test`/Widget 关键路径（与项目现状一致即可）。
4. **用例与优先级**：将验收标准转为用例条目（Given/When/Then 或 checklist）；标注 **P0 / P1**。
5. **运行与交付**：列出**建议**本地或 CI 执行方式（例如 `flutter test`、`integration_test` 等——以子项目 README 或既有脚本为准，若不确定则写「待开发确认具体命令」）；同一 PR/批次须能跑通规划中的自动化项。
6. **交接**：明确**由遵守 `developer-delivery-workflow` 的开发**实现自动化与补全命令；本 skill 产出作为**完成标准与缺口台账**。

## 与多角色总控的衔接

- 若上游已有 PM（`product-manager-workflow`）产出：以其中的验收标准与范围为主干映射用例。
- 若上游有移动端 UI/UX（`mobile-ui-ux-workflow`）产出：将**关键路径、屏幕清单、加载/空/错等状态**及文末「界面测试提示」纳入**界面层**用例与优先级。
- 若上游有架构（`software-architect-workflow`）产出：将「需开发验证」中与可测性、观测性、契约相关项纳入 API/界面策略。
- 在总控流水线中，本 skill 对应 **阶段四**（PM → 移动 UI/UX → 架构 → **测试策略** → 开发）。
- 不要求用户必须 `@` 多角色总控 skill；单独 `@` 本 skill 即可执行本工作流。

## 输出格式（固定结构，便于粘贴进 PR/任务单）

1. **测试范围摘要**：本期是否要求单元 / API / 界面（各是/否/降级及原因）。  
2. **用例清单**：按层级分列，含 P0/P1。  
3. **与验收标准的追溯**：可简表「验收条目 → 用例编号」。  
4. **执行与缺口**：建议命令或入口；无法自动化部分的原因与跟进。  
5. **任务清单建议句**：为新功能单列「单元测试 / API 测试 / 界面测试」三项勾选或子任务（与 `new-feature-testing-requirement.mdc` 衔接）。

## 输出习惯

- 用语简洁、可执行；避免空泛「要充分测试」类表述。  
- 交付说明可与 `delivery-template-rule.mdc` 中「测试与验收」一节对齐，便于复制。
