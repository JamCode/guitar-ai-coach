# 多角色交付总控工作流

## 用途与用法

- **用户只需 `@` 本 skill**（可附带需求描述或上下文）。
- 执行本工作流时，须**严格按阶段顺序**产出；每一阶段遵守**对应子 skill** 的角色边界与格式。子 skill 路径如下（相对仓库根目录）：
  - 阶段一：`.cursor/skills/product-manager-workflow/SKILL.md`
  - 阶段二：`.cursor/skills/mobile-ui-ux-workflow/SKILL.md`
  - 阶段三：`.cursor/skills/software-architect-workflow/SKILL.md`
  - 阶段四：`.cursor/skills/testing-strategy-workflow/SKILL.md`
  - 阶段五：`.cursor/skills/developer-delivery-workflow/SKILL.md`

## 数据流（谁吃谁的产出）

| 阶段 | 角色 | **输入（读谁）** | **输出（给谁用）** |
|------|------|------------------|-------------------|
| 一 | 产品经理 | 用户需求与上下文 | **阶段二至五**；核心是范围与**验收标准**。 |
| 二 | 移动端 UI/UX | **阶段一全文**（+ 用户补充情境/平台约束） | **阶段三、四、五**；把**屏幕、导航、关键状态与交互**交给架构对齐实现边界，交给测试映射界面用例，交给开发作界面实现依据。 |
| 三 | 软件架构师 | **阶段一全文** + **阶段二全文**（若未跳过；见下） | **阶段四、五**；契约、模块、非功能与可测性。 |
| 四 | 测试策略 | **阶段一全文**（必选）+ **阶段二全文**（若存在）+ **阶段三全文**（若存在） | **仅阶段五**：测试范围、用例清单、P0/P1、运行口径与缺口；开发按此落地自动化并遵守 `new-feature-testing-requirement.mdc`。 |
| 五 | 开发交付 | **阶段一** + **阶段二**（若有）+ **阶段三**（若有）+ **阶段四** + 用户即时指示 | 代码、配置、测试实现与提交推送（见 `developer-delivery-workflow`）。 |

- **阶段三与阶段二**：架构应以 **PM + 移动 UI/UX** 为依据，避免技术与屏幕流程脱节；若**跳过阶段二**，阶段三仅以阶段一 + 用户给定的界面/导航假设为准，须在阶段三开头**单列说明**所缺交互输入及风险。
- **阶段四与前置阶段**：移动 UI/UX 中的**关键路径与界面状态**应进入测试策略的界面层规划；架构产出支撑 **API/鉴权** 等策略。若**跳过阶段二**，阶段四须在开头说明界面侧依据（如仅 PM + 架构）；若**跳过阶段三**，阶段四仅以阶段一 + 阶段二 + 用户冻结的技术假设为准，并说明风险。
- **阶段五与阶段四**：开发实现功能与**自动化测试**时，以阶段四为**完成标准与任务拆分依据**；界面实现还须对齐阶段二（若存在）；不得在未说明理由的情况下静默省略阶段四已要求的测试层级。

## 上下文与子 skill 的加载

- 若当前对话上下文中**未包含**某一子 skill 的全文，在执行该阶段前须使用 **Read** 读取对应 `SKILL.md`，再按该文件执行。
- 不得凭记忆臆造子 skill 内容；边界与交接格式以文件为准。

## 阶段一：产品经理（必须先完成）

1. **完全按照** `.cursor/skills/product-manager-workflow/SKILL.md` 执行（复述、范围、验收、风险、输出模板等）。
2. 阶段一产出须**自成一体、可交接**：末尾包含该 skill 要求的 **「需要架构/开发回答的问题」**（最多 5 条）。
3. 本阶段**不写**技术方案、接口字段、代码与 DDL。

## 阶段二：移动端 UI/UX（以阶段一为输入）

1. **输入**：**阶段一（PM）全文**；可附用户补充的设备、深色模式、无障碍等约束。
2. **完全按照** `.cursor/skills/mobile-ui-ux-workflow/SKILL.md` 执行（导航、屏幕清单、状态与交互、需产品确认/需开发落实等）。
3. **不做**：不写业务代码；不替代 PM 扩写商业指标。
4. 若本期**无用户可见的移动端变更**（如纯后端、纯数据任务），可**跳过本阶段**，须在阶段三开头**一句话说明**跳过原因；阶段四、五以阶段一及后续可用输入为准。

## 阶段三：软件架构师（以阶段一 + 阶段二为输入）

1. **输入**：**阶段一全文**；**阶段二全文**（若未跳过）；可附用户补充技术约束。
2. **完全按照** `.cursor/skills/software-architect-workflow/SKILL.md` 执行。
3. 产出须包含该 skill 要求的 **「需产品确认」** 与 **「需开发验证」**；复杂决策可用 ADR 风格。
4. 本阶段**不替代**开发提交成片实现代码（除非用户明确要求写代码且与 `software-architect-workflow` 例外一致）。

## 阶段四：测试策略（以阶段一 + 阶段二 + 阶段三为输入）

1. **输入**：
   - **必选**：**阶段一（PM）全文**，尤其验收标准与范围；
   - **推荐**：**阶段二（移动 UI/UX）全文**（关键路径、加载/空/错等状态、可选「界面测试提示」）；**阶段三（架构）全文**（接口、鉴权、非功能与可测性）。
2. **完全按照** `.cursor/skills/testing-strategy-workflow/SKILL.md` 执行（分层映射、P0/P1、运行与缺口、轻量降级理由等）。
3. **不做**：默认不写测试实现代码；实现归属阶段五。
4. 若**已跳过阶段二或阶段三**，在本阶段开头说明依据与假设，并仍产出可交给阶段五的清单。

## 阶段五：开发交付（以阶段一至阶段四为输入）

1. **输入**：
   - **必选**：阶段一（PM）全文；**阶段四（测试策略）全文**；
   - **推荐**：阶段二（移动 UI/UX）、阶段三（架构）全文；按实际执行到的阶段**齐套引用**，缺失部分须由用户给定约定并在本阶段开头说明。
2. **完全按照** `.cursor/skills/developer-delivery-workflow/SKILL.md` 执行（含 pull、DDL 目录、增量功能与测试、提交推送等）。
3. 测试实现须**覆盖阶段四**已规划的层级与 P0 条目（除非阶段四已登记自动化缺口并给出本期手工验收步骤）；**界面实现**须对齐阶段二（若存在）。
4. 若需求或接口仍含糊，先按 `instruction-clarification-alignment.mdc` 澄清再改代码。

## 输出编排（单次回复或多轮均可）

- **推荐结构**：用清晰标题分隔五段，例如 `## 阶段一：产品经理`、`## 阶段二：移动端 UI/UX`、`## 阶段三：软件架构师`、`## 阶段四：测试策略`、`## 阶段五：开发交付`。
- **轻量需求**：若用户声明为「小改动 / 轻量变更」，可**跳过阶段三（架构）**；阶段二若**无界面变更**可一并跳过；阶段四仍应执行，但可按 `testing-strategy-workflow` 与 `new-feature-testing-requirement.mdc` 做**轻量降级并写明原因**；阶段五仍须遵守 `developer-delivery-workflow` 与仓库规则。

## 总控角色边界

- **做**：编排顺序、明确每阶段输入输出、确保子 skill 被遵守；在缺失上下文中主动 Read 子 skill。
- **不做**：用总控文案替代子 skill 的具体步骤；在阶段一抢写技术方案或在阶段三抢写完整 PRD（除非用户明确要求且与子 skill 一致）。
