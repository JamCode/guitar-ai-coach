# 和弦字典 — 产品需求

> 状态：草案（汇总讨论结论，待评审后锁定范围）  
> 关联能力：与现有「和弦进行 → 点击和弦查看指法」共用同一套 **和弦解释/指法** 能力，本页侧重 **主动查询** 与 **多按法展示**。

---

## 1. 背景与目标

### 1.1 用户痛点

用户在看谱时遇到**不认识的和弦符号**，不知道左手如何按弦，也缺少对和弦性质的快速理解。

### 1.2 产品目标

提供独立页面 **「和弦字典」**：用户**无需先「生成和弦进行」**，即可根据和弦符号获得 **多种常用按法** 与 **乐理说明**，降低查谱摩擦。

支持在 **不同调号下查看同一和声**（见 **§5.1**）：例如谱子是 D 调里的某个和弦，用户换到 G 调（或其它调）时，能同时看到 **变调后的和弦符号** 以及 **该调语境下的按法**（与「仅改符号、按法不变」区分开，见 5.1.3）。

### 1.3 成功标准（方向性）

- 用户能通过**以选择为主**的操作，在**较短时间**内找到与谱面一致或等价的符号，并看到**至少一种**可尝试的按法；理想情况下可对比 **多种按法** 选择顺手的一种。  
- 结果在 **调号、难度** 语境下与全站 explain 逻辑**可解释、可一致**（见下文待决项）。

### 1.4 从主页进入：入口与主页是否要改

**结论：要改，但宜「轻量」**——现在主页实质只有「生成和弦进行」一条主路径，若不增加**固定可见入口**，用户很难发现字典；不必重做整页布局。

#### 推荐做法（择一或组合，按实现成本从低到高）

| 方案 | 做法 | 优点 | 注意 |
| --- | --- | --- | --- |
| **A. 顶栏文案链接** | 在现有主卡片**上方**或**标题同一行**增加「和弦字典」文字链（或次要按钮），点击进入字典路由。 | 改动小、始终可见 | 需与现视觉层级协调，避免抢过主按钮 |
| **B. 双 Tab（顶或底）** | **底部 Tab** 更接近 App / PWA、单手友好（线框见 `home-prototype.html`）；顶栏 Tab 更偏桌面网页习惯。二者择一即可，实现上均为「同壳 + 路由/视图切换」。 | 心智清晰 | 要统一壳内「调号/难度」是否两 Tab 共用（建议**共用**，与 PRD §5、§7.2 一致） |
| **C. 主按钮下方次级链** | 「生成和弦」下增加一行小字「查单个和弦 → 和弦字典」。 | 零抢主 CTA | 曝光略弱，可配合 A |

#### 字典页返回

- 字典页需有 **返回主页 / 返回和弦进行**（文案与路由一致即可），避免用户进了子功能回不去。

#### 与线框原型的关系

- 原型 `chord-dictionary-prototype.html` 仅为字典页布局；**全站导航**可在下一版原型或实现时在 `App.vue` 增加顶栏/Tab。

#### 待决（可并入 §7）

- 使用 **Vue Router 独立路径**（如 `/dictionary`）还是 **Hash / 同页切换**；涉及分享链接与 PWA 时优先 **独立路径**。

## 2. 范围

### 2.1 MVP（第一版建议包含）


| 项         | 说明                                                                                  |
| --------- | ----------------------------------------------------------------------------------- |
| 独立入口      | 全站可进入「和弦字典」页面；**主页须有固定可见入口**（见 **§1.4**，推荐顶栏链接或双 Tab + 字典页返回主页）。 |
| **以选择为主** | **不以文本框为主路径**；通过 **分步点选** 拼出和弦符号（见第 3 节）。                                           |
| 符号预览      | 选择过程中**实时显示**将用于查询的符号预览（如 `Am7`、`G/B`），用户确认「与谱一致」后再发起查询。                            |
| **多按法列表** | 查询成功后，**列出多种**合理、可演奏的按法（见第 4 节），而非单张指法图。                                            |
| 单条说明结构    | 每条按法含：**指法图**、**短标签**（把位/开放/横按等）、**一两句按法说明**；和弦级可有**总述**（构成音、性质等，与现有 explain 语义对齐）。 |
| 免责声明      | 保留或强化「AI/指法因人而异、以实际音准为准」等说明。                                                        |
| 重新校准      | 需定义粒度：**整组和弦重算**或**单条按法重算**（见 4.4、6.3）。                                             |
| **调号切换（变调查看）** | 用户可在字典内切换 **目标调**（12 调）：**和弦符号**按产品统一的变调规则更新；**按法与说明**随目标调 **重新获取**（与 explain 的 `key` 一致）。可选区分「谱子原调」与「当前查看调」（见 **§5.1**）。 |


### 2.2 建议紧随 MVP 的增强（V1.5）


| 项       | 说明                                                 |
| ------- | -------------------------------------------------- |
| 常用和弦快捷区 | 一排高频 chip（如 C、G、Am、F、Dm、Cmaj7…），**一点即查**，覆盖多数看谱场景。 |
| 最近查询    | 本地保存最近 N 条（如 10～20），无需登录。                          |
| 复制      | 一键复制符号或摘要，便于分享。                                    |


### 2.3 明确暂不纳入 MVP（可列 backlog）

- 尤克里里 / 非标准调弦。  
- 变调夹（Capo）与谱面对照模式。  
- 完全离线、无 AI 的穷尽词库。  
- URL 带查询参数便于分享（可放 V1.5）。  
- 「高级」自由输入 / 剪贴板识别（若坚持零文本框，可长期不做；若接受逃生口，可折叠后置）。

## 3. 交互：以选择拼出和弦（主路径）

### 3.1 设计原则

- **主路径不得依赖大段文字输入**；通过 **chip / 单选 / 二级列表** 完成符号构建。  
- 与全站 **调号记法**（升号/降号习惯）保持一致，避免同一音两种写法混用导致用户困惑。

### 3.2 分步选择（推荐结构）


| 步骤          | 内容                                   | 说明                                       |
| ----------- | ------------------------------------ | ---------------------------------------- |
| ① 根音        | 12 音高之一（含升降记法，与产品统一规则）               | 升降可在主选一种，另一种放二级，减少一屏拥挤。                  |
| ② 和弦性质      | 大三、小三、属七、大七、小七、sus2/4、add9、dim、aug 等 | **MVP 仅弹唱高频子集**；其余收入「更多类型」或二级页，避免首版选项爆炸。 |
| ③ 延伸/变化（可选） | 初版可极简：默认「无」或仅最常见项                    | 后续再扩展 b9、#11 等爵士向符号。                     |
| ④ 低音/转位（可选） | 无转位 或 指定 slash 低音音名                  | 生成如 `G/B`；具体交互见 **待决项 7.1**。             |


### 3.3 常用和弦快捷区（强烈建议与分步选择同页或首屏可见）

- 展示 **高频和弦** 芯片，**单次点击**即带入该符号并触发查询（或先进入预览再确认，产品可二选一）。  
- 与分步选择 **互补**：简单和弦走快捷，复杂和弦走分步。

### 3.4 移动端与布局

- 根音、性质可用 **横向滑动 chip** 或 **底部列表**。  
- 步骤控制在 **3～4 步以内**，或使用 **单页分区 + 纵向滚动**，避免迷路。  
- 结果区：**纵向列表**展示多条按法；避免依赖横向拖动多张图才能看完核心信息（若采用横滑需单独写清规则）。

### 3.5 预览与确认

- 在发起请求前展示 **最终符号预览**（如 `Fmaj7`），用户 **确认查询** 后再请求后端/模型，减少误触与无效请求。

## 4. 多按法展示

### 4.1 用户价值

同一和弦符号存在 **多种合理按法**（开放、不同把位、横按、省略弦等）；仅展示一种容易与用户习惯或教材版本冲突。**字典页应以「多种常用按法」为默认体验。**

### 4.2 展示结构（每条按法一条卡片）

每条至少包含：

- **指法图**（与现有 ChordDiagram 语义一致：6→1 弦、frets、手指、横按等）。  
- **短标签**：如「开放把位」「某品附近」「横按」「省略低音弦」等，便于扫读。  
- **简短说明**：该按法适用场景（弹唱转换、避免闷音等）。  
- **可选**：难度/手型跨度提示（与全站「难度」概念对齐或独立标签，待实现时统一）。

和弦级可保留 **一段总述**（构成音、和弦结构说明），避免每条卡片重复长文。

### 4.3 「尽量都列出来」的验收口径（可落地）

完全穷举所有把位 **不纳入范围**。采用：


| 规则   | 说明                                                 |
| ---- | -------------------------------------------------- |
| 数量   | 单次查询返回 **N 种**常用按法，**N 建议默认 3～5**，**上限建议 8**（可配置）。 |
| 多样性  | 多种按法之间应 **视觉与把位上可区分**，避免多张近乎相同的图。                  |
| 优先级  | 优先覆盖：**开放 → 常见封闭/横按 → 高把位简化**；极复杂延伸和弦可适当减少条数。      |
| 扩展加载 | 若产品希望「更多」：可采用 **首屏 N 条 +「加载更多变体」**，控制单次延迟与成本。      |


### 4.4 与现有后端的关系（需求层约束）

当前 **POST /chords/explain** 形态为 **单 symbol → 单份 explain**（含缓存）。本需求需要 **同一符号对应多份指法**，实现上需其一或组合：

- 单次响应携带 **explain 数组**；或  
- 分次请求（如带 `variant` / `voicing_index`）；或  
- 首包主按法 + 异步拉取替代按法。

**缓存键**需覆盖「和弦 + 调 + 难度 + 变体集合或版本」，避免与单按法逻辑冲突；具体键设计由技术方案补充。

### 4.5 「重新校准」

- **建议**：支持对 **单条按法** 重新生成/校准，避免一条不满意就整组重算。  
- 若 MVP 仅能做整组重算，须在页面上说明行为差异。

## 5. 上下文：调号与难度

- 和弦解释接口现有入参包含 **key、level**，影响文案与缓存。  
- **待决**：字典页 **调号 / 难度** 是否与首页 **完全联动**（同一份全局状态），还是字典页 **独立默认值**（如默认 C 调、中级）。见 **7.2**。

### 5.1 按谱选调与变调查看（和弦名 + 按法）

**场景**：谱子在 **调 X** 下写了某个和弦，用户希望换到 **调 Y** 弹（或仅想搞清「在别的调里这个和声写成什么、怎么按」）。

#### 5.1.1 用户可见行为

- **和弦符号**：随 **目标调（查看调）** 按全站统一的 **半音移调规则** 更新（与「和弦进行」里变调展示一致；含 **slash** 时，低音与根音一并正确移位）。  
- **按法图与指法说明**：在 **目标调** 下 **重新生成/拉取**（`key` = 目标调），并仍支持 **多按法列表**（第 4 节）。  
  - *原因简述*：吉他上「同一和声」在不同调常用 **不同把位**；仅平移品位不能覆盖所有教学/弹唱习惯，故按法以 **目标调的 explain** 为准，而不是简单平移 frets。  
- **可选增强**：界面上同时标注 **谱调**（用户可选填或从选择器设定）与 **当前查看调**，避免「我以为还在原调」的混淆；若不做双调展示，至少通过 **当前选中的调号 chip** 明确「以下符号与按法均相对于该调」。

#### 5.1.2 与「变调夹」的区别

- **变调夹（Capo）**：谱面和弦名往往不变、实际音高整体上移；当前 **backlog**（见 2.3），勿与 **选调变调** 混为一谈。  
- **本节变调**：固定调记谱下 **和弦字母名随调改变**，按法按 **新调名** 解释。

#### 5.1.3 实现提示（供技术方案引用，非 UI 细则）

- **符号变调**：可复用现有 **`POST /chords/transpose`** 对「单行和弦符号」的变换能力（与进行页同一套规则）。  
- **说明与缓存**：每次 **目标调** 变化后，explain 请求应带 **新 `key`**；缓存键需包含 **symbol（变调后）+ key + level**（及多按法版本号，见 4.4）。

## 6. 验收标准（MVP 勾选用）

- 用户**不依赖主路径文本框**即可完成查询（允许「高级」折叠逃生口若后续另定）。  
- 选择完成后有 **符号预览**，确认后发起查询。  
- 对约定测试集（至少含：**大三/小三**、**一种七和弦**、**一种 slash 如 G/B**），返回 **≥2 种**可区分按法，且图与说明 **自洽**（含闷音弦时一致）。  
- 错误/非法组合有 **明确空态或错误提示**，不白屏。  
- 移动端核心内容 **可纵向滚动读完**，加载态可感知。  
- 免责声明可见；重新校准行为与 PRD 一致（整组或单条）。  
- [ ] **变调查看**：在至少两个不同目标调之间切换时，**和弦符号**符合统一变调规则；**按法**随目标调更新且与 **该调下** 的符号语义一致（可选取含 slash 的用例测一条）。

## 8. 附录：与现有功能的关系

- **和弦进行内点开和弦**：仍为「从上下文进入单和弦」；字典页为「无进行上下文」的入口。  
- **和弦进行变调**：进行列表用 **`/chords/transpose`** 变 **符号**；字典页 **变调查看** 复用同一套 **符号规则**，并额外按新调 **重拉 explain**（多按法），二者职责不同但规则应对齐。  
- **能力复用**：指法图组件、乐理说明风格、免责声明、（演进后的）explain 接口应对齐，降低用户认知成本。  
- **命名**：对外文案「和弦字典」；内部路由/接口名由实现约定。
