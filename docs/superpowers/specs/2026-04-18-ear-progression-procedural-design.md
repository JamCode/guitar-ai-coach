# 和弦进行练耳：规则生成出题器设计

**日期**：2026-04-18  
**分支**：`cursor/e5eaf664`（实现时以当时分支为准）  
**状态**：已评审意图；§7 和弦个数上限已拍板  

## 1. 背景与目标

- **现状**：Bank B（和弦进行）依赖 `ear_seed_v1.json` 题池抽题。
- **目标**：改为**运行时基于规则生成**题目，不依赖预置题池条目；保留与现有播放/判分链路一致的数据形态（`music_key`、`progression_roman`、四选一选项等）。
- **难度**：**初级 / 中级 / 高级** 三档，难度主轴为 **进行长度 + 和声复杂度（在保守符号体系内）**。
- **非目标**：本题不扩展罗马数字语法（见第 2 节）；不讨论 UI 大改版。

## 2. 符号与解析范围（用户选定：方案 A）

仅使用**大调自然音级**上的三和弦功能标记（与当前 `EarPlaybackMidi` 能力对齐）：

- 允许片段：`I`、`ii`、`iii`、`IV`、`V`、`vi`；需要减三和弦级时可用 **`vii`** 或 **`vii°`**（实现时与现有解析统一一种规范写法）。
- **禁止**在题面/正确答案字符串中引入：`7`、`maj7`、`bII`、借用、二级属等扩展标记（后续若单独立项可再开「扩展解析」规格）。

**说明**：中级/高级的难度**不**通过「写七和弦」提升，而通过 **和弦个数 N**、**调性覆盖**、**进行模板与干扰项易混度** 提升。

## 3. 硬约束（不变量）

1. **选项和弦个数一致**：设本题正确答案的 `progression_roman` 按 `-` 分割后段数为 **N**，则四个选项的标签经同样分割后 **必须均为 N**，且 **N 与音频和弦个数一致**。禁止「正确 3 和弦、某选项 4 和弦」类题目。
2. **播放与揭示**：生成结果必须可被现有 **`EarPlaybackMidi.forProgression`** / **`EarProgressionPlayback`** 消费（指法序列或 MIDI 回退路径行为与现网一致）。
3. **四选项互异**：四个选项的标签字符串**两两不同**；正确答案对应唯一 `correct_option_key`。
4. **可复现性**：生成器接受**伪随机源**（如 `RandomNumberGenerator` / 种子），单元测试在固定种子下输出确定。

## 4. 难度分档（与方案 C 对齐，且在方案 A 内）

| 档位 | 和弦个数 N | 调性策略 | 进行来源策略 | 干扰项策略（概要） |
|------|------------|----------|----------------|--------------------|
| 初级 | **3** | 小集合轮换（建议首版：`C, G, D, F`；实现可配置） | **经典短进行白名单**（如 `ii-V-I`、`I-IV-V`、`vi-IV-I` 等，表内条目均为 N=3） | 从「同 N=3 的其它白名单项」抽取；不足时用受控「单段替换」生成第二梯队候选并校验合法性与互异性 |
| 中级 | **4** | 在初级基础上适度扩调（建议增加 `A, E, Bb` 等，可配置） | **常见四和弦型白名单为主**（如 `I-V-vi-IV`、`vi-IV-I-V`、`I-vi-IV-V` 等，均为 N=4） | 同 N 异型 + 易混对；必要时单段替换，保证仍为大调自然级且 N 不变 |
| 高级 | **5**（**固定且封顶**：本 spec 下 **N 最大为 5**，不允许 6 及以上） | 更广的调性集合（可实现为「全自然大调调名」或接近） | **受限生成**：在白名单骨架上拼接 / 或带终止与流畅约束的短随机游走，**词汇仍限于 §2** | 优先选「与正确答案共享前缀或后缀」的同 N 标签，增强辨析难度 |

## 5. 推荐架构：混合生成器（模板为主 + 高级受限游走）

### 5.1 模块边界

- **`EarProgressionQuestionSpec`**（名称可调整）：描述单次生成的输出字段，与 `EarBankItem` 初始化所需字段对齐（`id`、`mode`、`question_type`、`prompt_zh`、`music_key`、`progression_roman`、`options`、`correct_option_key`、`hint_zh` 等；`id` 可用 `UUID` 或 `prog-{difficulty}-{timestamp}-{seq}`。
- **`EarProgressionProceduralGenerator`**：
  - 输入：`difficulty`、`rng`、可选「调性覆盖策略」配置。
  - 输出：`EarBankItem`（或 Spec + 适配器转 `EarBankItem`）。
- **常量表**：允许存在**小型白名单与易混表**（作为规则的一部分），**不等价于**用户可刷的 JSON 题池。

### 5.2 干扰项生成（满足 §3）

1. 从「当前难度、当前 N」的**错误标签候选集**中无放回随机抽取，直到凑满 3 个错误项。
2. 候选不足时：**单段替换**——随机选一位置，将该段替换为另一合法自然级数字符串，校验：  
   - 段数仍为 N；  
   - 整串不等于正确答案；  
   - 与已有选项不重复；  
   - 仍可由 `EarPlaybackMidi` 解析（若某组合解析失败则重试或换位置）。
3. 选项顺序：四键位 `A–D` 随机洗牌，**正确键位随机**，避免总出现在同一位置。

### 5.3 与宿主集成（实现阶段遵循）

- **Bank B 会话**：在 `maxQuestions == nil` 或带 cap 模式下，**下一题**时调用生成器而非从 `doc.bankB` 取条目（具体替换点以 `EarMcqSessionViewModel` 为准）。
- **Bank A / 其它**：不在本题范围。
- **资源**：不再要求为每题配置 `audio_ref`（若 `EarBankItem` 仍含可选字段，生成器留空即可）；播放走 SF2 / 品格路径与现逻辑一致。

## 6. 测试与验收

- **属性测试**：对大量随机种子，断言四选项分割段数均等于 N，且与 `progression_roman` 的 N 一致；正确答案唯一；选项两两不同。
- **黄金用例**：固定种子下快照 2–3 题的结构化字段（调性、罗马串、选项集），防止回归。
- **人工抽检**：高级档少量听辨，确认「游走」规则未产生过分怪异进行；若怪异率偏高，收紧高级转移表而非扩展罗马语法。

## 7. 和弦个数上限（已定稿）

- **初级 N=3、中级 N=4、高级 N=5**；全 spec 内 **`progression_roman` 按 `-` 分割后的段数 N 不得超过 5**（**禁止 N=6 或更长**，不再留配置开关延长）。
- **vii / vii°** 在白名单中的使用频率：首版可仅在高级或特定模板出现，避免初级过载（仍为建议，实现可微调）。

## 8. 范围外

- 小调、转调、借调听辨。
- 罗马数字扩展与和弦外音。
- 自动评估「和声进行优美度」的机器学习模型。

- **一致性**：难度由 N 与模板/游走定义；符号约束 §2 与 §4、§5.2 一致。
- **范围**：单文件规格仅覆盖 Bank B 规则生成；不包含具体 Swift 文件改动清单（留给 implementation plan）。
