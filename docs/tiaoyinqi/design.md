## 阶段一：产品经理

### 复述理解

用户希望把练习 Tab 里"和弦切换"从只能练 C-G-Am-F 一条进行，扩展为：  
1. 提供**多条常见和弦进行**供选择  
2. 支持 **12 个调**自由切换  
3. 可选**和弦复杂度**（三和弦 / 七和弦 / 高级）

### 用户与场景

吉他学习者（初学～进阶），想在特定调性、特定复杂度下练习常见和弦进行。  
典型场景：  
- 初学者在 C 调练三和弦 I-V-vi-IV  
- 中级者想在 G 调练七和弦 ii-V-I  
- 进阶者想用高级和弦练卡农进行

### 本期范围

| In Scope | Out of Scope |
|----------|-------------|
| 预置 ≥ 12 条常见和弦进行（级数定义） | 用户自创进行 |
| 12 调选择 | 节拍器/音频/MIDI 播放 |
| 3 档复杂度（基础/进阶/高级） | 后端同步（维持本地存储） |
| 选择页 UI + 预览和弦名 | "节奏扫弦""爬格子热身"改造 |
| 练习页顶部展示所选和弦序列 | 和弦指法图展示 |
| 练习记录保存调性/复杂度/进行信息 | |

### 验收标准

**P0（必须）**  
1. 用户可看到按风格分组的多条预置和弦进行列表  
2. 可选 12 调中任一调  
3. 可选 3 档复杂度中任一档  
4. 选择后进入练习页，顶部显示转调后的实际和弦名称  
5. 练习记录包含进行名称、调性、复杂度信息，可在历史中查看

**P1（建议）**  
1. 选择页实时预览和弦名称  
2. 卡农等长进行的和弦名自动换行  

### 风险与假设

- 假设现有 `DiatonicRoman` / `ChordTransposeLocal` 可复用于转调  
- 高级和弦命名可能偏复杂（如 maj9、m7b5），但吉他用户普遍可接受  
- 借用和弦（bVII/bVI/bIII）在升号调里的异名处理需自行实现

### 需架构/开发回答的问题

1. 转调引擎是复用 `ChordTransposeLocal` 还是独立实现？  
2. 预置进行数据是纯前端 const，还是后续可能走后端配置？  
3. `PracticeSession` 新字段的向后兼容方式？

## 阶段二：移动端 UI/UX

### 交互范围摘要

本期新增 1 个屏幕（和弦进行选择页），改动 1 个屏幕（练习计时页增加和弦展示），不改底部导航结构。

### 导航与流程

```
练习首页 → 点击"和弦切换·开始"
       → 和弦进行选择页（新）
              选进行 / 选调 / 选复杂度 / 预览
              → 点击"开始练习"
                     → 练习计时页（已有，顶部加和弦序列）
                            → 结束 → 弹窗打分 → pop 回选择页
                                           → 可换配置再练 / 返回首页
```

### 屏幕说明

| 屏幕 | 目标 | 主要控件 | 空/错/载 |
|------|------|---------|---------|
| 和弦进行选择页 | 选定进行+调+复杂度 | 进行列表(按风格分区 ListTile)、调性 ChoiceChip 横滑、复杂度 SegmentedButton、预览卡片、底部"开始练习"按钮 | 无网络依赖，不涉及加载/错误态 |
| 练习计时页（改） | 计时练习 | 顶部和弦序列 Wrap、现有计时器+操作按钮 | 同现有 |

### 关键交互

- 进行列表默认选中第一条  
- 调性默认 C  
- 复杂度默认"基础"  
- 三者任一变化，预览卡实时刷新和弦名  
- "开始练习"按钮在有效选择时始终可点  
- 计时页顶部和弦序列仅展示，不可交互  

### 需产品确认

- 和弦进行的顺序是否需要用户可排序？→ 本期不做  

### 建议纳入界面测试的状态与路径

1. 打开选择页 → 默认选中第一条进行 + C 调 + 基础  
2. 切换调性/复杂度 → 预览和弦名变化  
3. 点击"开始练习" → 进入计时页且顶部展示和弦  
4. 计时 → 结束 → 保存 → pop 回选择页

## 阶段三：软件架构师

### 上下文

- Flutter 本地 App，练习模块无后端依赖  
- 已有 `DiatonicRoman`（级数→MIDI）、`ChordTransposeLocal`（和弦符号变调）、`ChordSelectCatalog`（12 调列表）  
- 练习数据存 SharedPreferences

### 推荐方案

**纯前端 const 数据 + 本地转调引擎**

| 模块 | 职责 |
|------|------|
| `ChordProgression` 模型 | id / name / romanNumerals / style |
| `ChordComplexity` 枚举 | basic / intermediate / advanced |
| `ChordProgressionEngine` | 级数×调×复杂度 → 和弦符号列表；内置 C 调查表 + 转调 |
| `kChordProgressions` 常量列表 | ≥ 12 条预置进行 |

**转调策略**：在 C 调为每个级数定义三档和弦符号（查表），再用半音差转到目标调。借用和弦（`b` 前缀级数）强制使用降号命名，避免升号调产生如 D# 代替 Eb 的问题。

**数据兼容**：`PracticeSession` 新增 3 个可空字段（`progressionId`/`musicKey`/`complexity`），`fromJson` 缺失时取 null，旧数据无损。

### 落地顺序

1. 数据模型 + 引擎 → 单元测试  
2. Session 模型扩展 → 测试  
3. 选择页 UI → Widget 测试  
4. 练习页改造 → Widget 测试  
5. 集成冒烟

### 用例清单

**单元层（P0）**  
- U1: `resolveChordNames('I-V-vi-IV', 'C', basic)` → `['C','G','Am','F']`  
- U2: `resolveChordNames('I-V-vi-IV', 'G', basic)` → `['G','D','Em','C']`  
- U3: `resolveChordNames('ii-V-I', 'C', intermediate)` → `['Dm7','G7','Cmaj7']`  
- U4: 借用和弦 `bVII` 在升号调使用降号命名  
- U5: PracticeSession 新字段 JSON 往返 + 旧数据缺失字段兼容  

**界面层（P0）**  
- W1: 打开选择页 → 进行列表非空、默认选中、预览卡显示和弦  
- W2: 切换调性 → 预览和弦变化  
- W3: 切换复杂度 → 预览和弦变化  
- W4: 点"开始练习" → 进入计时页 → 顶部显示和弦  
- W5: 完整流程：选择 → 计时 → 结束 → 保存 → 返回选择页
