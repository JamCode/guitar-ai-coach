# 本地离线扒歌（文件导入）设计说明

## 1. 需求目标与范围

### 目标

- 将现有“实时扒歌”方向调整为“**文件导入后、本地离线识别和弦**”。
- 首版优先验证：**能稳定导入文件、完成本地识别、在播放器界面中按时间轴展示和弦与指法**。
- 不依赖网络；商业授权可用于闭源 App 上架。

### 本期范围（In Scope）

- 输入来源：`相册`、`文件 App`
- 支持格式：
  - 音频：`mp3`、`m4a`、`wav`、`aac`
  - 视频：`mp4`、`mov`
- 单文件时长限制：**6 分钟以内**
- 本地离线识别：
  - 媒体解码与抽音轨
  - 音频预处理
  - ONNX 本地推理
  - 和弦时间轴生成
  - 原调识别
- 结果页能力：
  - 顶部和弦轨道
  - 和弦指法小卡片
  - 中间波形
  - 中间固定播放线
  - 底部进度条 + 播放/暂停
  - 播放时当前和弦高亮
  - 拖动进度条时高亮立即跳转
- 历史记录：
  - 自动保存到独立的“扒歌历史”
  - 可再次打开播放和查看结果
  - 支持删除历史记录
- 删除旧的实时扒歌入口与页面

### 非本期范围（Out of Scope）

- 实时麦克风扒歌
- 联网识别 / 云端模型
- 一键变调
- BPM、小节线、段落结构分析
- capo 自动建议
- 用户手动校对和弦
- 系统分享入口
- 6 分钟以上文件支持
- 更多格式（如 `flac`、`ogg`、`webm`、`mkv`）

## 2. 方案选型与结论

### 推荐方案

首版采用：**`consonance-ACE` + `ONNX Runtime Mobile` + `AVFoundation` + `Accelerate/vDSP`**

### 选型原因

- **能扒和弦**：`consonance-ACE` 是现成和弦识别模型，适合本地离线推理
- **能本地跑**：`ONNX Runtime Mobile` 可在 iPhone 上本地运行模型
- **商业授权可行**：
  - `consonance-ACE`：MIT
  - `ONNX Runtime`：MIT
  - Apple 原生框架：系统自带
- **格式支持足够实用**：本期 6 种格式均可通过 `AVFoundation` 处理，不需要引入 FFmpeg
- **未来可演进**：后续若需要提升精度或切换模型，可沿用同一运行时与结果数据结构

### 明确不采用

- `Essentia`：AGPL，不适合本项目闭源商业分发目标
- `Chordino / NNLS-Chroma`：GPL，不适合本项目闭源商业分发目标
- `CREMA`：可行但不是首选；当前方案在商业授权、iOS 落地与后续维护上更稳
- FFmpeg：本期格式范围内不需要，避免增加包体和集成复杂度

## 3. 原型与交互确认

### 原型说明

本次设计已确认采用“**播放器式结果页**”，而不是纯列表式结果页。

### 关键交互确认点

- 结果页采用“顶部和弦轨道 + 指法 / 中间波形 / 中间播放线 / 底部简洁播放控件”的结构
- 顶部不做“波形 / 和弦”切换
- 底部不放多余按钮，只保留：
  - 进度条
  - 播放 / 暂停
- 文件导入后不是边识别边播放，而是：
  1. 先完成本地识别
  2. 再进入结果页播放联动
- 播放到某一时刻时，当前和弦卡片高亮
- 拖动进度条时，和弦高亮立即跳转

### UI/UX 风格约束

- 新增的扒歌入口页、处理中页、结果页、历史页，**视觉风格必须与 `swift_ios_host` 当前已有页面保持一致**
- 不另起一套独立的播放器皮肤，不做与现有 App 脱节的“纯黑专业音频工具风”
- 颜色、层级、圆角、留白、导航栏与按钮样式，优先复用当前 Swift 端已有主题体系与组件习惯
- 实现上优先沿用现有主题与样式能力（如 `SwiftAppTheme`、现有 page background / card / button 风格），而不是为扒歌页单独发明一套视觉规范
- 本次确认的“波形 + 和弦轨道 + 指法”是**信息结构与播放器布局参考**，不是要求照搬外部 App 的视觉皮肤
- 最终效果应满足：把新扒歌页放到当前 Swift App 里，看起来仍属于同一个产品

## 4. 信息架构与页面

### 4.1 扒歌入口页

用途：发起文件导入，并展示最近历史入口。

页面元素：

- 标题：本地离线扒歌
- 主操作：
  - 从相册导入
  - 从文件导入
- 说明文案：
  - 支持 `mp3 / m4a / wav / aac / mp4 / mov`
  - 单文件最长 6 分钟
- 最近历史入口（展示最近若干条）

### 4.2 处理中页

用途：反馈当前识别状态，降低等待焦虑。

页面元素：

- 文件名
- 步骤状态：
  - 提取音轨
  - 本地识别和弦
  - 整理结果
- 取消按钮

说明：

- 不强调复杂技术细节
- 不承诺“快速识别”，重点是让用户知道系统仍在工作

### 4.3 结果详情页

用途：作为首版核心使用页面，边播放边看和弦与指法。

页面结构：

- 顶部信息：
  - 文件名
  - 原调
  - 已保存状态
- 顶部和弦轨道：
  - 按时间顺序展示和弦段
  - 每个和弦附带指法小卡片
  - 当前播放位置对应的和弦高亮
- 中间主区域：
  - 大波形
  - 固定中线表示当前播放位置
- 底部控件：
  - 时间进度条
  - 播放 / 暂停按钮

交互规则：

- 识别完成后才进入本页
- 点击播放后，音频开始播放，当前和弦随时间推进高亮
- 拖动进度条后：
  - 媒体跳转到新位置
  - 波形位置变化
  - 当前和弦立即重新计算并高亮

视觉要求：

- 页面背景、导航栏、标题层级、按钮样式、卡片圆角与边框语义需与当前 Swift 页面统一
- 波形区域虽然是主视觉，但整体观感仍应与现有 App 的卡片、分区和主题色一致
- 和弦卡片与指法小卡片优先复用现有和弦相关页面的视觉语言，避免出现完全不同的一套图形表达

### 4.4 扒歌历史页

用途：承载历史结果，不与“我的谱”混用。

页面元素：

- 历史列表
- 每条记录展示：
  - 文件名
  - 原调
  - 识别时间
- 操作：
  - 点击进入结果详情页
  - 删除历史记录

## 5. 技术设计

### 5.1 模块划分

#### 1）文件导入层

负责从相册或文件 App 获取媒体文件，并做输入合法性校验。

职责：

- 选择文件
- 校验文件扩展名
- 校验时长是否 <= 6 分钟

#### 2）媒体解析层

基于 `AVFoundation` 将输入统一成识别引擎可消费的音频数据。

职责：

- 音频文件直接读取
- 视频文件抽取音轨
- 统一输出为标准音频流 / PCM 数据

#### 3）识别引擎层

负责本地离线和弦识别。

技术组成：

- 预处理：`Accelerate / vDSP`
- 推理运行时：`ONNX Runtime Mobile`
- 识别模型：`consonance-ACE`

职责：

- 音频切片
- 特征预处理
- 本地模型推理
- 输出初始和弦片段

#### 4）结果整理层

负责把模型结果整理成 UI 可直接消费的结构。

职责：

- 合并过碎片段
- 计算原调
- 生成时间轴片段
- 生成“播放位置 -> 当前和弦”的映射

#### 5）历史存储层

负责保存和读取本地识别历史。

职责：

- 创建历史记录
- 查询历史列表
- 打开详情
- 删除记录

### 5.2 数据结构（最小集合）

#### `TranscriptionHistoryEntry`

- `id`
- `sourceType`：`photoLibrary | files`
- `fileName`
- `storedMediaPath`
- `durationMs`
- `originalKey`
- `createdAt`
- `segments[]`

#### `TranscriptionSegment`

- `startMs`
- `endMs`
- `chord`

说明：

- 指法图不落库存储，展示时根据 `chord` 关联现有和弦字典数据
- 为保证历史记录可再次打开并播放，导入完成后需要将原始媒体文件复制到 App 私有目录，并记录 `storedMediaPath`
- 波形数据采用“轻量摘要缓存”策略：识别完成后保存一份用于展示的波形摘要数据，详情页直接读取，避免每次打开历史都重新全量分析媒体

### 5.3 播放联动规则

- 播放器当前时间为 `currentTimeMs`
- 遍历 `segments[]`，找到满足：
  - `startMs <= currentTimeMs < endMs`
- 将该片段对应和弦卡片标记为当前
- 当用户拖动进度条时，同步更新 `currentTimeMs`
- 高亮切换与播放进度保持一致

## 6. 错误处理

首版统一做“人话提示”，不暴露技术异常。

### 错误类型

1. 文件格式不支持
2. 文件超过 6 分钟
3. 音轨解析失败
4. 本地识别失败
5. 识别结果为空

### 提示文案建议

- `当前只支持 mp3、m4a、wav、aac、mp4、mov`
- `当前只支持 6 分钟以内的文件`
- `这个文件的音轨读取失败，请换一个文件再试`
- `这次没能稳定识别出和弦，换一个更清晰的文件再试试`

## 9. 验收标准

### 功能验收

- 用户可以从相册或文件 App 导入支持格式文件
- 不支持格式会被明确拦截
- 超过 6 分钟的文件会被明确拦截
- 识别完成后可进入结果页
- 结果页可显示：
  - 原调
  - 和弦轨道
  - 指法小卡片
  - 波形
- 播放时当前和弦会高亮
- 拖动进度条时，高亮立即跳转
- 结果会自动保存到“扒歌历史”
- 历史记录可以再次打开和删除

### 体验验收

- 结果页整体像播放器，而不是纯文本列表
- 首版播放器控件保持简洁，不出现与目标无关的复杂编辑功能
- 旧实时扒歌入口不再对用户可见

### 质量验收

- 目标识别准确率：约 70% 水平，可满足首版可用性验证
- 常见流行歌曲场景下，结果应具备基本参考价值

## 10. 风险与后续建议

### 当前风险

- 本地模型在不同类型音源上的稳定性需要通过样本回归测试验证
- 视频文件音轨质量差时，识别结果可能波动
- 波形展示与播放高亮联动需要保证性能流畅

### 后续建议

- 二期可考虑补充：
  - 一键变调
  - BPM / 小节线
  - 用户手动校对和弦
  - 系统分享入口
  - 云端高精度识别作为增强模式
