# 节奏扫弦：八分最小单位与时值建模设计（A 版扩展）

> 分支：`cursor/48e3c1e5`  
> 日期：2026-04-21  
> 关联需求：扫弦展示改六线谱，并支持“下=一拍 / 下上=一拍”这类教材时值表达

## 1. 背景与问题

当前扫弦模式主要用固定 `cells[8]`（`1 & 2 & 3 & 4 &`）表达，每格仅包含 `down/up/rest`。  
这能表示方向，但不能直接表达教材常见语义：

- 某动作占一整拍（例如：`下` 一拍）
- 同一拍内两个动作（例如：`下上` 一拍）
- 带“拍/切音”（`x`）与空拍的分组口令

用户明确希望把“方向”与“时值”同时表达清楚，避免“看起来像对了，数拍却不对”。

## 2. 目标

1. 将扫弦模式的最小时间单位统一为 **八分音符（1/2 拍）**。
2. 支持“动作 + 时值”表达：例如 `下(1拍)`、`下(半拍)+上(半拍)`。
3. 在六线谱 A 版 UI 中让用户可看懂“哪个动作持续一拍、哪个是半拍动作”。
4. 兼容现有页面逻辑与历史数据，不引入破坏性改动。

## 3. 范围

### In Scope

- 4/4 拍号下的扫弦时值模型；
- 动作种类：`down / up / rest / mute`；
- UI 以八分网格显示（`1 & 2 & 3 & 4 &`）；
- 每条 pattern 可由事件流展开为 8 格展示。

### Out of Scope（本期不做）

- 6/8、3/4 等其它拍号；
- 十六分最小单位；
- 复杂连音线、三连音视觉记谱；
- 自动音频/节拍器联动。

## 4. 数据模型（推荐方案 A）

新增事件流结构（概念层）：

- `StrumActionKind`: `down | up | rest | mute`
- `StrumActionEvent`:
  - `kind: StrumActionKind`
  - `units: Int`（八分单位，`1=半拍`，`2=一拍`）

在 `StrummingPattern` 中增加：

- `events: [StrumActionEvent]`
- `meterNumerator: Int`（先固定 `4`）
- `meterDenominator: Int`（先固定 `4`）
- `smallestUnit: .eighth`（先固定）

兼容字段：

- 保留 `cells: [StrumCellKind]` 作为展示/兼容层，且由 `events` 派生生成。

## 5. 约束与校验

对 4/4 + 八分最小单位，定义：

- 每小节总单位数应为 `8`。
- 每个 `event.units >= 1`。
- `events.reduce(units) == 8` 必须成立，否则判为无效模式（测试与运行时保护）。

示例：

- `下(2), 下(1), 上(1), 空(2), 下(1), 上(1)`  
  单位和：`2+1+1+2+1+1 = 8`（合法）

## 6. 六线谱 A 版渲染规则（结合时值）

弦序固定：

- 顶部 `①`，底部 `⑥`（最粗弦在最下）

方向映射（已确认）：

- `down -> ↑`
- `up -> ↓`
- `rest -> ·`
- `mute -> ×`

时值显示：

- 若 `units=1`：占一个八分格，显示动作符号。
- 若 `units=2`：第一格显示动作符号，第二格显示“延续占位”（淡点或短横，不重复动作符号）。
- `units>2`（理论可用）：第一格动作 + 后续格延续占位（本期主要用 `1/2` 两种）。

## 7. 教材口令对齐（表达层）

每个模式额外给可读口令（用于说明卡片）：

- 例如：`下（一拍） | 下上（一拍） | 空（一拍） | 下上（一拍）`

要求：

- 口令与 `events` 一致；
- 口令与六线谱展示一致；
- 不允许“名字/口令/网格”三者冲突。

## 8. 测试策略

### 单元测试（P0）

1. `events -> cells` 展开正确（包括 `units=2` 的占位）。
2. 总单位数校验：非 8 单位应失败或被拒绝。
3. 映射规则固定：`down=↑`, `up=↓`, `rest=·`, `mute=×`。
4. 口令与事件长度一致性（最少覆盖核心样例）。

### 手工验收（P0）

1. 用户可看出“下是一拍”与“下上是一拍”的区别。
2. 六线谱弦序与箭头方向符合约定（①上⑥下，↓/↑映射不反）。
3. 切换“下一组”时时值展示不乱、不溢出。

## 9. 风险与控制

- **风险：模型升级影响面扩大**  
  控制：保留 `cells` 兼容层，分步迁移。
- **风险：延续占位样式不直观**  
  控制：第一版使用简单统一占位（淡点/短横），后续再做视觉优化。
- **风险：既有模式定义不规范**  
  控制：上线前统一清理 pattern 库并补测试快照。
